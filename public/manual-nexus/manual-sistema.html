⚙️Administración del Sistema

Módulo central de configuración de TOPS Nexus: gestión de roles, permisos, usuarios, centros de costo, plantillas y comunicados del sistema. Acceso restringido a perfiles con autorización explícita.

🔐
Módulo de acceso restringido

La sección Sistema solo está disponible para los roles Director y Administración. Cualquier cambio en permisos o roles puede afectar el acceso de todos los usuarios al sistema. Proceder con precaución y siempre con autorización del Director.

El módulo Sistema y Administración es el núcleo de gobierno de TOPS Nexus. Desde acá se define quién puede ver qué, quién puede hacer qué, cómo se clasifican los gastos contablemente y cómo se configura la integración con servicios externos como Mapbox para el tracking de flota.

Es la capa de seguridad y configuración que sustenta todo lo demás. Un cambio incorrecto aquí puede dejar usuarios sin acceso o, peor, dar acceso indebido a módulos sensibles como Compras o Finanzas.

👤 ¿Quién puede acceder?

El acceso a la sección Sistema está controlado por el permiso sistema.view (para ver) y sistema.admin (para administrar). Solo dos roles del sistema tienen estos permisos:

Rol sistema.view sistema.admin Qué puede hacer
🏆 Director ✓ ✓ Acceso total: ver, crear, editar y eliminar roles, usuarios, centros de costo y toda la configuración
📊 Administración ✓ — Puede ver la configuración del sistema pero no modificar roles ni permisos
📦 Operaciones — — Sin acceso a Sistema
💼 Comercial — — Sin acceso a Sistema
🏭 Depósito — — Sin acceso a Sistema
🔍 Auditor — — Sin acceso a Sistema (el auditor tiene solo lectura en otros módulos)
⚠️
Distinción importante: ver vs. administrar

El rol Administración tiene sistema.view pero NO tiene sistema.admin. Esto significa que puede consultar la configuración (roles, usuarios, centros de costo) pero no puede crear, editar ni eliminar nada en este módulo. Para cualquier cambio estructural en el sistema, es obligatorio el rol Director.

🔑 Sistema de Roles y Permisos (RBAC)

TOPS Nexus implementa un modelo de control de acceso basado en roles (RBAC, Role-Based Access Control). Esto significa que los permisos se asignan a roles, no directamente a usuarios. Cada usuario tiene un rol, y ese rol determina qué puede ver y hacer en el sistema.

ℹ️
¿Cómo funciona el RBAC en TOPS Nexus?

Al iniciar sesión, el sistema carga los permisos del rol del usuario (boot flags). Estos flags determinan qué secciones aparecen en el sidebar, qué botones son visibles y qué acciones están habilitadas. Todo esto ocurre en tiempo real al cargar la aplicación.

Estrategia de visibilidad (Estrategia B)

El sistema RBAC de TOPS Nexus aplica la seguridad en dos niveles:

  • Nivel dominio: oculta secciones completas del sidebar si el usuario no tiene acceso al módulo (ej: un Depósito no ve "Sistema" en el menú).
  • Nivel ítem: dentro de una sección visible, oculta opciones individuales según los permisos específicos (ej: dentro de Compras, oculta el botón "Firmar OC" si el usuario no tiene compras.sign).

Los flags de boot más relevantes para el módulo Sistema son:

  • canViewSistema — habilita la sección Sistema en el sidebar
  • canViewExecutive — acceso a vistas ejecutivas del Cockpit
  • canViewRrhhDocs — visibilidad de documentación de RRHH

🎭 Los 6 roles del sistema

TOPS Nexus tiene 6 roles predefinidos que cubren todas las necesidades operativas de la empresa:

🏆 Director
slug: director
22/22 permisos
Máxima autoridad del sistema. Único rol con capacidad de firma de Órdenes de Compra (compras.sign). Acceso completo a todos los módulos incluyendo la administración del sistema.
📊 Administración
slug: administracion
21/22 permisos
Equipo financiero y administrativo. Tiene acceso a casi todo excepto la firma de OC (compras.sign), que es exclusiva del Director. Puede ver el módulo Sistema pero no administrarlo.
📦 Operaciones
slug: operaciones
Permisos operativos
Coordinación de depósitos y servicios. Accede a módulos de servicios, WMS y tracking. Sin acceso a módulos financieros ni a la administración del sistema.
💼 Comercial
slug: comercial
Permisos CRM
CRM y pipeline de ventas. Accede al módulo Comercial y Cockpit. Sin acceso a compras, finanzas ni administración del sistema.
🏭 Depósito
slug: deposito
Permisos básicos
Operarios de depósito. Picking, recepción de mercadería y firma de órdenes de servicio. Acceso limitado a los módulos necesarios para la operación diaria del depósito.
🔍 Auditor
slug: auditor
Solo lectura
Rol de auditoría con acceso de solo lectura a todos los módulos. Puede ver pero nunca crear, editar ni eliminar información. Ideal para revisiones de control interno.
⚠️
Permiso crítico: compras.sign

El permiso compras.sign (firma de Órdenes de Compra) es exclusivo del rol Director. Ningún otro rol puede activarlo, independientemente de los demás permisos asignados. Esto es una restricción de seguridad intencional. Si una OC requiere firma y el Director no está disponible, el proceso debe quedar en pausa hasta su disponibilidad.

🔑 Gestión de Roles

Desde /settings/roles podés ver, crear y editar los roles del sistema. Esta sección requiere el permiso sistema.admin (solo Director).

Ver un rol existente

Cada rol tiene su propia página de detalle en /settings/roles/[slug] donde se puede ver:

  • Nombre y slug del rol
  • Descripción y propósito
  • Lista completa de permisos habilitados por módulo
  • Usuarios que tienen este rol asignado actualmente

Crear un nuevo rol

➕ Pasos para crear un rol desde cero
▼
  1. Navegar a /settings/roles
  2. Hacer clic en el botón "Nuevo rol" (visible solo con sistema.admin)
  3. Completar el nombre del rol (ej: "Supervisor de Depósito")
  4. El slug se genera automáticamente desde el nombre. Se puede personalizar (solo letras minúsculas, guiones bajos, sin espacios)
  5. Agregar una descripción clara del propósito del rol
  6. Seleccionar los permisos por módulo usando las casillas de verificación
  7. Guardar el rol
  8. Ir a Usuarios para asignar este rol a las personas que lo requieran
⚠️
Principio de mínimo privilegio

Al crear un rol nuevo, comenzar siempre con el mínimo de permisos necesarios. Es más fácil agregar permisos después que revocarlos una vez que el usuario ya está trabajando con ellos.

✏️ Editar un rol existente
▼

Para editar los permisos de un rol existente:

  1. Ir a /settings/roles y seleccionar el rol a editar
  2. Usar el botón "Editar" en la página de detalle del rol
  3. Agregar o quitar permisos según se necesite
  4. Guardar los cambios
  5. Los cambios se aplican de forma inmediata para todos los usuarios con ese rol (al recargar su sesión)
🔴
Atención: cambios con impacto inmediato

Editar un rol afecta a todos los usuarios que lo tienen asignado. Si quitás un permiso a un rol que muchas personas usan, esas personas perderán acceso a esa funcionalidad de forma inmediata. Coordinar los cambios fuera del horario operativo cuando sea posible.

👥 Gestión de Usuarios

Desde /settings/users se gestiona la asignación de roles a los usuarios del sistema. Acá no se crean cuentas (eso depende del proveedor de autenticación), sino que se asocia cada cuenta con un rol de TOPS Nexus.

Asignar un rol a un usuario

  1. Ir a /settings/users
  2. Buscar el usuario por nombre o email
  3. Hacer clic en el usuario para ver su perfil
  4. En el selector de rol, elegir el rol apropiado
  5. Confirmar el cambio
  6. El usuario verá los nuevos permisos la próxima vez que inicie sesión

Cambiar el rol de un usuario

El proceso es idéntico a la asignación. Al cambiar el rol, el usuario pierde todos los permisos del rol anterior y gana los del nuevo. Si el usuario está actualmente logueado, los cambios se aplicarán al recargar la sesión.

💡
Un usuario, un rol

TOPS Nexus implementa un modelo de rol único por usuario. Cada persona tiene exactamente un rol asignado. Si alguien necesita permisos de múltiples roles, hay que crear un rol personalizado que combine los permisos necesarios, o revisar si el rol existente más cercano puede ampliarse.

🏛️ Centros de Costo

Los centros de costo son etiquetas contables que permiten clasificar los gastos e ingresos por área o proyecto dentro de la empresa. Son utilizados en el módulo de Compras y en Servicios para imputar correctamente cada transacción.

❓ ¿Para qué sirven los centros de costo?
▼

Permiten responder preguntas como:

  • ¿Cuánto gastó el área de Operaciones este mes?
  • ¿Qué porcentaje del gasto corresponde al depósito de Palermo vs. el de Tigre?
  • ¿Cuánto costó el proyecto X en total?

Sin centros de costo, todos los gastos se acumularían sin diferenciación, imposibilitando el análisis financiero por área.

➕ Crear un nuevo centro de costo
▼
  1. Ir a /settings/centros-costo
  2. Hacer clic en "Nuevo centro de costo"
  3. Ingresar el código del centro (ej: CC-001, OPS-DEP)
  4. Ingresar el nombre descriptivo (ej: "Depósito Central", "Administración", "Comercial")
  5. Opcionalmente, agregar una descripción
  6. Guardar. El centro de costo estará disponible de inmediato en Compras y Servicios.
🔗 ¿Cuándo se usa el centro de costo?
▼

Al crear o editar una:

  • Orden de Compra (OC): se requiere seleccionar el centro de costo al que se imputará el gasto.
  • Orden de Servicio (OS): permite identificar qué área o proyecto genera el servicio.

El campo de centro de costo suele ser obligatorio en Compras para garantizar la trazabilidad contable completa.

📍 Configuración de Tracking

La sección de Tracking en /settings/tracking permite configurar la integración con Mapbox para el seguimiento de la flota de vehículos.

🗺️ Configuración de Mapbox
▼

Para configurar la integración con Mapbox:

  • Token de acceso (API Key): ingresar el token público de Mapbox. Este token habilita los mapas y la geolocalización en tiempo real.
  • Estilo del mapa: se puede seleccionar entre el mapa estándar, satelital o dark mode para la vista de flota.
  • Centro del mapa: coordenadas de inicio (latitud/longitud) para centrar la vista por defecto.
⚠️
Token de Mapbox

El token de Mapbox es una clave de API que tiene asociados los límites de uso y la facturación. No compartirlo ni publicarlo. Renovarlo periódicamente según las políticas de seguridad de la empresa.

🚛 Configuración de flota
▼

La configuración de flota incluye:

  • Registro de vehículos activos (dominio, descripción, tipo)
  • Asignación de dispositivos GPS a vehículos
  • Zonas geográficas de operación

Esta funcionalidad está en etapa de diseño activo. Consultar con el equipo de Sistemas para el estado actual de implementación.

📋 Plantillas de Órdenes de Servicio

Las plantillas de OS (accesibles desde /templates) son modelos predefinidos para órdenes de servicio recurrentes. Permiten crear nuevas OS a partir de una estructura ya configurada, reduciendo el tiempo de carga y minimizando errores.

❓ ¿Cuándo usar una plantilla?
▼

Las plantillas son útiles para servicios que se repiten frecuentemente con las mismas características:

  • Mantenimiento preventivo mensual de equipos
  • Servicios de limpieza periódicos
  • Revisiones técnicas estándar
  • Cualquier OS que se genere más de 2-3 veces por mes con la misma estructura
➕ Crear una nueva plantilla
▼
  1. Ir a /templates
  2. Hacer clic en "Nueva plantilla"
  3. Definir el nombre de la plantilla (descriptivo, ej: "Mantenimiento mensual - Refrigeración")
  4. Completar los campos base: proveedor habitual, descripción del servicio, centro de costo
  5. Agregar los ítems estándar de la OS
  6. Guardar la plantilla
  7. Para usar la plantilla al crear una OS nueva, seleccionarla desde el botón "Usar plantilla" en la creación de OS
✏️ Editar y gestionar plantillas
▼

Desde la lista de plantillas se puede:

  • Editar una plantilla existente para actualizarla con nuevos datos
  • Duplicar una plantilla como base para una nueva variante
  • Archivar plantillas que ya no se usan (no se eliminan para mantener el historial)

Las plantillas son compartidas por todos los usuarios con permiso de creación de OS.

📢 Comunicados del Sistema

Los comunicados (/sistema/comunicados) son avisos que aparecen en el sistema para todos los usuarios. Se usan para informar sobre mantenimientos programados, cambios importantes en procesos, o cualquier novedad relevante.

📝 Publicar un comunicado
▼
  1. Ir a /sistema/comunicados
  2. Hacer clic en "Nuevo comunicado"
  3. Definir el tipo de comunicado: Informativo, Mantenimiento, Urgente o Crítico
  4. Escribir el título (breve y claro)
  5. Redactar el contenido del comunicado
  6. Seleccionar la fecha de publicación y, opcionalmente, una fecha de expiración
  7. Publicar. El comunicado será visible para todos los usuarios en el sistema
🏷️ Tipos de comunicados
▼
  • Informativo: novedades generales, nuevas funcionalidades, cambios de proceso. Aparece como banner azul.
  • Mantenimiento: avisos de interrupciones programadas. Aparece como banner amarillo.
  • Urgente: situaciones que requieren atención inmediata. Aparece como banner naranja.
  • Crítico: incidentes en curso, datos afectados, acciones requeridas de todos. Aparece como banner rojo.

💡 Casos frecuentes

🆕 Nuevo empleado: dar acceso al sistema
  1. Confirmar con RRHH y el responsable del área qué rol corresponde al nuevo empleado
  2. Asegurarse de que el empleado ya tenga cuenta de acceso al sistema (proceso fuera de TOPS Nexus)
  3. Ir a /settings/users
  4. Buscar al usuario por nombre o email
  5. Asignarle el rol correspondiente
  6. Verificar con el empleado que puede ingresar y que ve las secciones correctas
  7. Registrar el cambio en el legajo del empleado (RRHH)
🔄 Cambio de rol: empleado cambia de área o cargo
  1. Recibir la solicitud formal del cambio (email de RRHH o del Director)
  2. Ir a /settings/users y buscar al usuario
  3. Cambiar el rol al nuevo que corresponde
  4. Notificar al empleado del cambio y verificar que pueda acceder a las nuevas secciones
  5. Si hay acceso a datos sensibles que ya no corresponde (ej: pasaba de Director a Comercial), confirmar que ya no puede ver lo que no debe
🏛️ Agregar un nuevo centro de costo para un proyecto
  1. Recibir la solicitud de Contabilidad o del responsable del proyecto
  2. Definir el código y nombre del centro de costo (coordinar con Contabilidad para seguir la nomenclatura existente)
  3. Ir a /settings/centros-costo y crear el nuevo centro
  4. Comunicar a los usuarios de Compras y Servicios que ya está disponible
👋 Baja de empleado: revocar acceso
  1. Recibir la comunicación de baja de RRHH
  2. Revocar o deshabilitar la cuenta del empleado en el proveedor de autenticación (fuera de TOPS Nexus)
  3. En /settings/users, verificar que el usuario ya no pueda ingresar
  4. Registrar la baja en el legajo del empleado
🔴
Acción inmediata requerida

La revocación de acceso ante una baja debe realizarse el mismo día, idealmente antes del último día laboral del empleado. No esperar.

🔴 Errores comunes

🚫 "No tenés permisos para ver esta sección"
El usuario intenta acceder a un módulo para el que su rol no tiene el permiso view.
Solución: verificar el rol del usuario en /settings/users. Si el acceso es legítimo, asignar el rol correcto o crear un rol personalizado con el permiso necesario. Si no es legítimo, informar al usuario que debe solicitar acceso formalmente.
🚫 Usuario no aparece en la lista de usuarios
El usuario existe pero no aparece en /settings/users para asignarle un rol.
Solución: verificar que el usuario haya iniciado sesión al menos una vez en TOPS Nexus. El sistema registra a los usuarios cuando se loguean por primera vez. Si nunca ingresó, pedirle que lo haga antes de intentar asignarle un rol.
🚫 Usuario con rol correcto pero sin ver la sección
El rol tiene los permisos correctos pero el usuario no ve la sección en el menú.
Solución: pedirle al usuario que cierre sesión y vuelva a ingresar. Los boot flags de permisos se cargan al iniciar sesión; si el rol fue cambiado mientras la sesión estaba activa, se necesita un nuevo login para aplicar los cambios.
🚫 No se puede firmar la OC ("Firma no disponible")
El botón de firma de Orden de Compra no aparece o está deshabilitado.
Solución: compras.sign es un permiso exclusivo del rol Director. Si el usuario que necesita firmar no tiene el rol Director, no puede hacerlo. No es un error del sistema; es una restricción intencional. La OC debe ser firmada por el Director.
🚫 El centro de costo no aparece en el selector de Compras
Al crear una OC o OS, el centro de costo recién creado no aparece en el desplegable.
Solución: verificar que el centro de costo fue guardado correctamente en /settings/centros-costo. Si fue creado hace muy poco, intentar recargar la página del módulo de Compras.

🛡️ Buenas prácticas de seguridad

  • 🔒
    Principio de mínimo privilegio. Asignar siempre el rol con los permisos mínimos necesarios para la tarea del usuario. No dar rol Director o Administración a alguien que solo necesita operar en Depósito. Si hay duda, empezar con el rol más restrictivo y escalar solo si hay necesidad probada.
  • 📋
    Auditar el listado de usuarios periódicamente. Al menos una vez por trimestre, revisar la lista de usuarios y verificar que todos tienen el rol correcto. Los roles cambian cuando las personas cambian de función y no siempre se actualiza en tiempo real.
  • 🔍
    Usar el rol Auditor para revisiones. Cuando un externo necesita revisar información del sistema (consultor, auditor contable), asignarle el rol Auditor. Tiene solo lectura de todos los módulos activos, sin posibilidad de modificar nada.
  • ⏱️
    Revocar accesos de forma inmediata. Ante una baja, renuncia o cambio de área, revocar el acceso el mismo día. No esperar al proceso burocrático completo para quitar los permisos.
  • 📝
    Documentar los cambios de rol. Registrar quién autorizó el cambio de rol de cada usuario. Esto facilita las auditorías futuras y ayuda a resolver disputas sobre accesos.
  • 🔐
    Proteger el permiso sistema.admin. Solo el Director debería tener sistema.admin. No ceder este permiso a roles de administración general. Es el permiso más crítico del sistema porque permite modificar todos los demás permisos.
  • 📢
    Coordinar cambios de roles masivos. Si necesitás editar un rol que tiene muchos usuarios asignados, coordinar el cambio fuera del horario laboral o con previo aviso a los afectados. Los cambios son inmediatos y pueden interrumpir el trabajo en curso.
  • 🔑
    Rotar las claves de API periódicamente. El token de Mapbox y cualquier otra clave de integración configurada en Sistema debe renovarse según la política de seguridad de la empresa. Nunca compartir estas claves por canales inseguros.

✅ Checklist: Incorporar un nuevo usuario

Seguir estos pasos cada vez que se incorpora un nuevo miembro al equipo. Hacer clic en cada ítem para marcarlo como completado.

  • 01 Confirmar con RRHH el nombre completo, email y fecha de inicio del nuevo empleado
  • 02 Confirmar con el responsable del área qué rol de TOPS Nexus corresponde al nuevo empleado
  • 03 Coordinar con IT/Sistemas la creación de la cuenta de acceso (usuario y contraseña inicial)

❓ Preguntas frecuentes (FAQ)

❓ ¿Puedo crear roles personalizados?
▼

Sí. El sistema permite crear roles completamente personalizados desde /settings/roles/new. Se puede definir cualquier combinación de permisos. Esto es útil si los 6 roles estándar no se ajustan a alguna necesidad particular de la empresa (ej: un "Supervisor de Depósito" con más permisos que Depósito pero menos que Operaciones).

❓ ¿Un usuario puede tener más de un rol?
▼

No. TOPS Nexus implementa un modelo de un usuario, un rol. Si alguien necesita permisos de múltiples áreas, la solución es crear un rol personalizado que combine todos los permisos necesarios. Esto mantiene el modelo de seguridad simple y auditable.

❓ ¿Qué pasa si modifico los permisos de un rol mientras los usuarios están trabajando?
▼

Los cambios de permisos de rol se aplican al recargar la sesión. Los usuarios que tienen sesión activa continuarán con los permisos anteriores hasta que cierren sesión y vuelvan a entrar, o hasta que recarguen la página. En algunos casos el sistema puede detectar el cambio y forzar una recarga. Para cambios críticos, recomendamos hacerlos fuera del horario operativo.

❓ ¿El Auditor puede ver los datos de Sistema y Administración?
▼

No. El rol Auditor tiene solo lectura en los módulos operativos (Compras, Servicios, Comercial, etc.) pero no tiene acceso al módulo Sistema. Esto es intencional: un auditor externo no debería poder ver la configuración de seguridad interna del sistema. Si se necesita que un auditor revise la configuración de roles, debe hacerse con supervisión del Director.

❓ ¿Se pueden eliminar los 6 roles base del sistema?
▼

Los 6 roles base (director, administracion, operaciones, comercial, deposito, auditor) son roles del sistema y no deben eliminarse. Son los pilares del modelo de seguridad. Sin embargo, sus permisos pueden editarse si hay una necesidad justificada. Cualquier cambio a un rol base debe ser autorizado explícitamente por el Director y documentado.

❓ ¿Las plantillas de OS se comparten entre todos los usuarios?
▼

Sí. Las plantillas de OS son compartidas por todos los usuarios que tienen acceso al módulo de creación de órdenes de servicio. No son plantillas personales; cualquier usuario autorizado puede verlas y usarlas. La creación y edición de plantillas requiere permisos específicos del módulo Sistema.

❓ ¿Cómo sé qué rol tiene asignado un usuario?
▼

Desde /settings/users, buscar al usuario y ver su perfil. El rol asignado aparece claramente. Los usuarios también pueden ver su propio rol en la sección "Mi Espacio" de su perfil.

❓ ¿Los comunicados del sistema son permanentes?
▼

No. Al crear un comunicado se puede definir una fecha de expiración. Una vez pasada esa fecha, el comunicado deja de mostrarse automáticamente. Los comunicados sin fecha de expiración quedan activos indefinidamente hasta que se archiven manualmente.

📖 Glosario técnico

Términos clave del módulo Sistema y del modelo de seguridad de TOPS Nexus:

RBAC
Role-Based Access Control. Modelo de seguridad donde los permisos se asignan a roles, y los roles a usuarios. Permite gestionar el acceso de forma centralizada y escalable.
Rol
Conjunto predefinido de permisos que define qué puede hacer un usuario en el sistema. Cada usuario tiene exactamente un rol asignado.
Slug
Identificador único de un rol en formato texto simple (sin espacios ni caracteres especiales). Ej: director, operaciones. Es el identificador interno que usa el sistema.
Permiso
Autorización específica para realizar una acción en un módulo. Se expresa como modulo.accion. Ej: compras.view, sistema.admin.
Módulo
Sección funcional del sistema con su propio conjunto de permisos. Ej: Compras, Servicios, Sistema, Comercial.
Acción
Operación concreta que se puede realizar dentro de un módulo. Las acciones disponibles son: view, create, edit, delete, sign, export, admin.
Boot flags
Variables de configuración que se calculan al iniciar sesión y determinan qué secciones y funcionalidades son visibles para el usuario. Se derivan de los permisos del rol.
sistema.admin
Permiso crítico que permite administrar la configuración del módulo Sistema (roles, usuarios, centros de costo). Exclusivo del rol Director en la configuración estándar.
compras.sign
Permiso de firma de Órdenes de Compra. El único permiso que es exclusivo del rol Director. Ningún otro rol puede tenerlo activado.
Centro de costo
Etiqueta contable que identifica el área o proyecto al que se imputa un gasto o ingreso. Permite el análisis financiero por departamento o proyecto.
Plantilla OS
Modelo predefinido para una Orden de Servicio recurrente. Contiene la estructura base (proveedor, ítems, centro de costo) para agilizar la creación de nuevas OS.
Principio de mínimo privilegio
Buena práctica de seguridad que indica asignar siempre el conjunto mínimo de permisos necesarios para que el usuario pueda hacer su trabajo. Reduce el riesgo de accesos indebidos.
Mapbox
Proveedor de mapas y geolocalización utilizado por TOPS Nexus para el módulo de Tracking de flota. Requiere una clave de API (token) configurada en Sistema.
canViewSistema
Boot flag que habilita la sección Sistema en el sidebar de navegación. Se activa cuando el rol del usuario tiene al menos el permiso sistema.view.
