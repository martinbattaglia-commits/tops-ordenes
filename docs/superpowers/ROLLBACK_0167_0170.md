# ROLLBACK — Migraciones F4.3 `0167`–`0170` (Tareas + Workflows)

> Reversa COMPLETA de la capa DB de F4.3. Ejecutar SOLO con autorización expresa
> de Dirección, a mano en el SQL Editor de prod (`arsksytgdnzukbmfgkju`), en
> ORDEN INVERSO al apply (0170 → 0169 → 0168 → 0167). Ninguna migración de F4.3
> modifica objetos previos de forma destructiva.

## Consideraciones previas

- **Datos**: `drop table connect_tasks` destruye tareas reales. Evaluar backup
  (`create table connect_tasks_bak as select * from connect_tasks;` + followers
  + instances) o export CSV antes.
- **Residuos aceptados** (inofensivos): valores de enum `'task'`
  (`connect_conversation_kind_t`) y `'task_admin'` (`permission_action_t`) — 
  Postgres NO soporta DROP VALUE; filas de `audit_log`
  (`entity='connect_task'`); notificaciones `kind='connect_task'` ya emitidas
  (el fallback de hrefFor las manda a `/connect/notificaciones`); conversaciones
  `kind='task'` huérfanas (siguen siendo hilos normales del motor Connect);
  eventos `knowledge_events` solo si la fuente se hubiera habilitado (no
  esperado: D-F43-9 la deja apagada).
- El frontend se revierte re-deployando el commit anterior; DB y front degradan
  con gracia por separado (patrón verificado en F4.2; la CollabCard se
  auto-oculta si las tablas no existen).


## 3. Revertir 0168 (schema + permiso)

```sql

 using public.permissions p
 where rp.permission_id = p.id and p.slug = 'connect.task_admin';
delete from public.permissions where slug = 'connect.task_admin';

-- Realtime (idempotente).
do $$ begin
  execute 'alter publication supabase_realtime drop table public.connect_tasks';
```
