# Plan Global E2 — Knowledge F0.5.2 (Worker asíncrono multi-fuente)

> **For agentic workers:** REQUIRED SUB-SKILL: cada subfase (E2.0–E2.4) se ejecuta con un plan detallado propio (`superpowers:writing-plans` → `subagent-driven-development`) autoredactado al inicio de esa subfase. Este documento es el **plan global / de arquitectura**: congela componentes, contratos, migraciones, validaciones, riesgos y la compatibilidad con E3. No contiene los pasos TDD bite-sized de cada subfase (esos viven en el plan de la subfase).

**Estado:** BORRADOR PARA APROBACIÓN G7 de Dirección. No se escribe código, no se crean migraciones, no se toca prod hasta la aprobación.

**Goal:** Activar el procesamiento **asíncrono** del Knowledge Layer (cola/outbox + worker + estados + retry) y **extenderlo a múltiples fuentes**, con KPIs, panel administrativo fail-closed y observabilidad propia (EOL), 100% aditivo sobre E1 y sin romper contratos públicos.

**Architecture:** Mantiene la pila aprobada SoR → SoK (event-sourced read-model) del Master Dossier. E1 dejó la estructura de outbox **inerte** (columnas `status/retry_count/available_at/processed_at/error` + `dispatch_idx` ya existen en `knowledge_events`, pero todo nace `processed`). E2 la **enciende**: el emisor acepta `p_status`, un worker drena `pending/failed` vía un endpoint cron fail-closed, y nuevas fuentes se suman por el Adapter Pattern + Source Registry ya probados con `audit_log`.

**Tech Stack:** PostgreSQL 17 (Supabase prod `arsksytgdnzukbmfgkju`) · PL/pgSQL `SECURITY DEFINER` · Next.js App Router (route handlers + server components) · TypeScript (`src/lib/knowledge/`) · GitHub Actions (cron) · RBAC `has_permission` + RLS `visibility_key`.

## Global Constraints (verbatim, aplican a TODA tarea de E2)

- **Aditividad estricta:** ninguna migración de E1 (`0125–0131`) se reabre ni se modifica. Todo es objeto nuevo o `create or replace` aditivo backward-compatible.
- **Contratos públicos congelados:** `knowledge_event_canonical` (composite type, 13 campos) **NO cambia**. `v_knowledge_timeline` / `v_knowledge_entity_360` / `KnowledgeEvent` / `mapTimelineRow` mantienen su forma. La firma de `knowledge_emit_event` solo crece por el final con un parámetro con default.
- **Hardening obligatorio (lección H-E1-1):** toda función `SECURITY DEFINER` nueva o recreada lleva `set search_path = public, pg_temp` + `revoke all from public` + **`revoke execute from anon, authenticated`** (explícito; el `revoke from public` NO alcanza por los default privileges de Supabase) + `grant execute to service_role`.
- **Numeración:** prod rastrea por timestamp; el prefijo `NNNN` es del worktree. E2 usa **`0132+`** (verificado libre en todas las ramas; máx actual `0131`). Verificar en vivo antes de cada aplicación.
- **Aplicación:** archivo-por-archivo vía MCP `apply_migration` sobre `arsksytgdnzukbmfgkju` (project_id verificado por operación), una migración por vez, validación read-only automática, STOP ante inconsistencia, Run Log por subfase. (Estándar E1.)
- **D-1 = `staff`** ya aplicado en `0127`; los entes compras/proveedores/flota/compliance proyectan `staff` (no `public_auth`).
- **Permisos RBAC existentes (0129, verificados):** `knowledge.view`, `knowledge.create`, `knowledge.edit`, `knowledge.delete`, `knowledge.admin`. **No hay** `knowledge.manage_sources` ni `knowledge.export`; el panel admin se gatea con **`knowledge.admin`**.
- **Fuera de alcance E2:** Búsqueda Universal (→ F0.5.3, decisión Dirección), Centro de Monitoreo, OIL, Memoria Operativa, KIL/MACL. Nada de pgvector/embeddings (`knowledge_documents/chunks` quedan scaffold inerte).

## 1. Arquitectura

### 1.1 Componentes y responsabilidades

| Componente | Tipo | Responsabilidad | Límite (lo que NO hace) |
|---|---|---|---|
| **Emisor `knowledge_emit_event`** | RPC SECDEF (existe, se extiende) | Único punto de escritura en `knowledge_events`. Acepta el estado inicial (`p_status`). | No conoce fuentes; no decide cadencia; no hace fan-out. |
| **Cola / Outbox** | columnas en `knowledge_events` (existen, se activan) | Persistir el ciclo de vida del evento (`status/retry_count/available_at/processed_at/error`). | No es tabla nueva; no es broker externo. |
| **Dispatcher RPC** | RPC SECDEF (nuevo, E2.1) | Reclamar lote `pending/failed` (lock), transicionar estados, aplicar backoff. | No ejecuta el efecto de negocio (lo hace el worker/handler). |
| **Worker `/api/knowledge/drain`** | Route handler (nuevo, E2.1) | Orquestar: reclamar lote → ejecutar fan-out → marcar resultado. Fail-closed por `CRON_SECRET`. | No corre lógica de UI; no expone datos sin auth. |
| **Automatización** | GitHub Actions cron (nuevo, E2.1) | Disparar `/api/knowledge/drain` en intervalo. | No procesa; solo dispara. |
| **Adaptadores de fuente** | `project_<x>()` trigger + `backfill_<x>()` RPC (nuevos, E2.2) | Mapear filas de una tabla fuente a `knowledge_event_canonical` y emitir vía el emisor. | No insertan directo en `knowledge_events`; no tocan el pipeline. |
| **Source Registry** | tabla `knowledge_sources` (existe) | Catálogo + gate `enabled` por fuente. | No contiene lógica por fuente. |
| **KPIs** | vistas/RPC de lectura (nuevos, E2.3) | Exponer backlog, throughput, fallos, dead-letter, lag de backfill. | No mutan estado; agregados, no PII. |
| **Panel admin** | páginas `(app)/knowledge/admin/*` (nuevo, E2.3) | UI staff fail-closed: ver KPIs, gestionar `enabled` de fuentes, disparar backfill/drain manual. | No escribe `knowledge_events` directo; todo vía RPC. |
| **EOL (observabilidad propia)** | `observability.ts` + catálogo (existe/se reconcilia, E2.4) | Eventos técnicos en canal separado + `correlation_id` + structured logging. | NO escribe en `knowledge_events`; no monta dashboards externos. |

### 1.2 Contratos públicos (estables tras E2)

- **`knowledge_emit_event(p_event knowledge_event_canonical, p_status text default 'processed') → uuid`** — la firma crece por el final; las 1..N llamadas actuales (trigger/backfill de audit_log) siguen compilando sin cambios.
- **`knowledge_event_canonical`** (composite, 13 campos) — **inmutable** en E2.
- **`v_knowledge_timeline` / `v_knowledge_entity_360`** (`security_invoker`) — forma estable; E2 puede agregar columnas al final sin romper `mapTimelineRow`.
- **`knowledge_sources(source_table, enabled, …)`** — contrato del registry; E2 agrega filas, no columnas que rompan.
- **`public.notifications`** — destino del fan-out del worker; E2 **consume/extiende**, no rompe (clave para E3/Connect).

### 1.3 Dependencias

```
E2.0 (p_status)  ──►  E2.1 (worker/cola/estados/cron)  ──►  E2.2 (adaptadores)  ──►  E2.3 (KPIs+panel)  ──►  E2.4 (EOL)
```

## 2. Subfases

### E2.0 — Contrato `p_status` del emisor  *(OBLIGATORIO, primero — prerequisite #1)*
- **Objetivo:** que un evento pueda nacer `pending` sin romper a los emisores síncronos actuales.
- **Entregables:** migración `0132` (overload aditivo de `knowledge_emit_event` con `p_status text default 'processed'` + re-hardening del nuevo overload); índice de timeline (prereq #4).
- **Definition of Done:** `knowledge_emit_event(event,'pending')` inserta una fila `status='pending'` que aparece en `dispatch_idx`; las llamadas existentes (sin `p_status`) siguen insertando `processed`; ACL del nuevo overload = `{postgres, service_role}`; 0 cambios en `knowledge_event_canonical`; smoke de regresión: backfill de audit_log sigue dando idempotencia 0 en 2ª corrida.

### E2.1 — Worker + Cola + Estados + Automatización
- **Objetivo:** drenar `pending/failed` con transiciones de estado, retry y backoff, disparado por cron, fail-closed.
- **Entregables:** migración `0133` (dispatcher RPCs: reclamar lote con lock, marcar `processed/failed/dead`, backoff); `src/app/api/knowledge/drain/route.ts` (GET/POST, `CRON_SECRET`, `?dry=1`); `.github/workflows/knowledge-drain.yml`; `src/lib/knowledge/drain.ts` (capa data).
- **Definition of Done:** un evento `pending` se transiciona a `processed` al drenar; un fallo transitorio → `failed` con `retry_count++` y `available_at` futuro (backoff); excedido `MAX_RETRIES` → `dead`; el endpoint responde 401 sin `Bearer CRON_SECRET`; `?dry=1` no muta; lock evita doble-procesamiento; el cron dispara y reporta. Sin tocar contratos de lectura.

### E2.2 — Adaptadores de nuevas fuentes
- **Objetivo:** sumar fuentes al timeline (orden sugerido por valor/seguridad: `recon` → `purchase_order`/compras → `tesoreria` → `custody` → `rrhh`), con degradación grácil.
- **Entregables:** migración(es) `0134(+)` con, por fuente: `knowledge_<src>_to_canonical()` (mapeo), `project_<src>()` (trigger AFTER INSERT, defensivo), `knowledge_backfill_<src>(p_limit)` , seed en `knowledge_sources`. Fix del sentinel `'∅'` en el mapeo (prereq #5) y reconciliación del vocabulario `entity_type` del CASE (prereq #15). Opcional: helper genérico de backfill (prereq #13).
- **Definition of Done:** por cada fuente activada, sus filas históricas se materializan (backfill idempotente: 2ª corrida = 0) y sus filas nuevas se proyectan en vivo; `enabled=false` corta la proyección (gate); una fuente ausente (`to_regclass` NULL) no rompe; `visibility_key` correcto por fuente (compras/flota/compliance = `staff` por D-1); ningún `entity_id` colisiona bajo `'∅'`.

### E2.3 — KPIs + Panel administrativo (fail-closed)
- **Objetivo:** observabilidad operativa del pipeline para staff, y control de fuentes/backfill/drain bajo RBAC.
- **Entregables:** migración `0135` (vistas/RPC de KPIs: backlog por estado/fuente, throughput, dead-letter, lag de backfill; + fix `set search_path` en `has_permission` prereq #8); páginas `(app)/knowledge/admin/*` gateadas por `knowledge.admin`; wrapper TS de `entity_360` con filtro obligatorio (prereq #7); keyset pagination en `listTimeline` (prereq #6); enlace en Sidebar (dominio "Conocimiento", gateado).
- **Definition of Done:** un usuario sin `knowledge.admin` ve `AccesoRestringido` (fail-closed); el panel muestra KPIs reales y permite togglear `enabled` y disparar backfill/drain **vía RPC** (no INSERT directo); `entity_360` nunca se consulta sin `WHERE entity_type+entity_id`; la paginación profunda usa cursor por `seq`; `has_permission` tiene `search_path` fijo.

### E2.4 — Observabilidad propia (EOL)
- **Objetivo:** catálogo único de eventos técnicos y `correlation_id` end-to-end, conforme ADR-ENG-1/D20.
- **Entregables:** catálogo único `nombre → capa → emisor → receptor` (prereq #9) como fuente de verdad; reconciliación de literales TS (`observability.ts`) ↔ SQL (emisor/triggers/worker); emisión real de structured logs en worker/adaptadores con `correlation_id` por lote; contratos de métricas KPI alineados.
- **Definition of Done:** todo evento técnico del worker/adaptadores usa un nombre del catálogo único (0 divergencias TS↔SQL); `correlation_id` se propaga app→RPC→trigger→worker; los logs salen al canal técnico (stdout/Postgres log), **nunca** a `knowledge_events`; documentado el contrato de métricas.

## 6. Compatibilidad con E3 — Nexus Link

> E2 se diseña también desde la óptica de Nexus Link (Connect, capa SoX que consume SoK). **No se implementa** la integración; solo se garantiza que E2 no genere restricciones para E3 y deje listos los puntos de reúso.

### 6.1 Decisiones de E2 que preparan Nexus Link
- **Outbox + worker + estados (E2.0/E2.1):** Connect tiene su propio outbox de mensajes/incidentes (spec: `connect_outbox` con el mismo ciclo pending→processing→processed/failed/dead). El patrón dispatcher + `/api/.../drain` + cron de E2 es la **plantilla** que Connect reutiliza para su fan-out de menciones/notificaciones.
- **Adapter Pattern + Source Registry (E2.2):** Connect se registra como una **fuente más** (`ConnectAdapter`: mensajes/incidentes → timeline) sin tocar el pipeline. E2 deja el patrón probado y multi-fuente.
- **Fan-out a `public.notifications` (E2.1):** Connect **extiende** `public.notifications` (no la duplica). E2 debe consumir/escribir notificaciones **sin romper su contrato**, para que Connect construya encima.
- **EOL (E2.4):** el catálogo único de eventos técnicos y `correlation_id` end-to-end son la base que los componentes de Connect deben seguir (ADR-ENG-1).
- **KPIs + panel admin (E2.3):** patrón de panel staff fail-closed reutilizable por la administración de Connect.

### 6.2 Contratos que E3 reutilizará (deben quedar estables/compatibles)
- `knowledge_emit_event(p_event[, p_status])` — Connect emite sus eventos por aquí.
- `knowledge_event_canonical` — Connect mapea sus mensajes/incidentes a este contrato.
- `knowledge_sources` (registry) — Connect se da de alta como fuente.
- `v_knowledge_timeline` / `v_knowledge_entity_360` — Connect consume el timeline (chat contextual sobre entidades).
- `public.notifications` — contrato de notificaciones compartido.
- Patrón cron fail-closed (`CRON_SECRET`) + dispatcher RPC con lock.

### 6.3 Componentes que permanecen estables (E2 no los rediseña; E3 los hereda)
- El emisor único y el Source Registry (sin condicionales por fuente — OCP).
- El modelo `visibility_key` (`public_auth`/`staff`/`client:`/`perm:`) y D-1.
- La capa de lectura TS (`listTimeline`, `mapTimelineRow`) — E3 lee el mismo read-model.

### 6.4 Interfaces públicas a mantener compatibles para E3
- Firma del emisor (solo crece por el final).
- Composite `knowledge_event_canonical` (inmutable).
- Vistas de consumo (`security_invoker`; columnas solo se agregan al final).
- Contrato de `public.notifications`.
- Endpoint/auth de worker (`Bearer CRON_SECRET`) como convención de crons internos.

> **No se hace en E2:** tablas/migraciones/UI/lógica de Connect; `connect_*`; integración con Cockpit; mensajería. Eso es E3, en su propia sesión y plan.
