# Plan Detallado E2.1 — Worker / Cola / Estados / Reintentos / Automatización

> Sub-skill de ejecución (post-G7): `superpowers:subagent-driven-development`. Este documento es el plan detallado de E2.1; la implementación bite-sized se ejecuta tras la aprobación G7.

**Estado:** BORRADOR PARA APROBACIÓN G7. No se escribe código, no se crean migraciones, no se toca prod hasta la aprobación.

**Goal:** Construir el motor operativo (worker) que drena los eventos `pending`/`failed` de `knowledge_events`, gestiona su ciclo de estados con reintentos/backoff, se dispara por cron cada 5 min vía un endpoint fail-closed, y es idempotente, concurrency-safe y recuperable ante fallos.

**Architecture:** Outbox ya provisto por E1/E2.0 (`knowledge_events.status/retry_count/available_at/processed_at/error` + `dispatch_idx`). E2.1 agrega: (1) RPCs `SECURITY DEFINER` de despacho (claim con lock, mark processed/failed/dead, recover stuck) en `0133`; (2) un orquestador TS de drenado; (3) el endpoint `/api/knowledge/drain` fail-closed; (4) el cron de GitHub Actions cada 5 min. El **efecto de negocio** del procesamiento (fan-out a notificaciones, etc.) NO es de E2.1: el procesador es un punto de extensión con default no-op; los handlers reales llegan en fases posteriores.

**Tech Stack:** PostgreSQL 17 (PL/pgSQL `SECURITY DEFINER`, `FOR UPDATE SKIP LOCKED`) · Next.js App Router route handler (`nodejs` runtime) · TypeScript (`src/lib/knowledge/`) · GitHub Actions cron.

## Global Constraints (heredadas; aplican a toda tarea de E2.1)

- **No modificar E1 ni E2.0.** `knowledge_events` (constraint de `status`, columnas), `knowledge_emit_event` (firma de `0132`) y `knowledge_event_canonical` quedan intactos.
- **Estado terminal de éxito = `processed`** (valor del `CHECK` de `0126`). El "completed" de la directiva mapea a `processed`; NO se altera el constraint.
- **Hardening obligatorio (H-E1-1):** toda función `SECURITY DEFINER` nueva: `set search_path = public, pg_temp` + `revoke all from public` + `revoke execute from anon, authenticated` + `grant execute to service_role`.
- **Fail-closed:** el endpoint exige `Authorization: Bearer ${CRON_SECRET}`; sin secreto válido → 401. Patrón existente (`src/app/api/compliance/sync/route.ts`).
- **Idempotencia y no-pollution:** los smokes que emiten eventos de prueba corren en transacción + `ROLLBACK` (probado en E2.0: el MCP lo honra), sin residuo en `knowledge_events` (append-only).
- **Numeración:** E2.1 usa **`0133`** (siguiente libre; verificar en vivo antes de aplicar).
- **Fuera de E2.1:** adaptadores de fuentes, KPIs, panel admin, catálogo EOL completo, Nexus Link. (El worker NO ejecuta efectos de negocio reales todavía.)

## 2. Arquitectura del Worker

```
GitHub Actions cron (*/5)  ──HTTP POST + Bearer CRON_SECRET──►  /api/knowledge/drain (route handler)
                                                                      │  (fail-closed auth, ?dry)
                                                                      ▼
                                                         src/lib/knowledge/drain.ts (orquestador)
                                                                      │
                          ┌───────────────────────────────────────────┼───────────────────────────────────┐
                          ▼                                            ▼                                   ▼
          knowledge_claim_batch(limit)                 processEvent(event)  [procesador no-op E2.1]   knowledge_mark_*()
          (UPDATE … FOR UPDATE SKIP LOCKED              → ok | error                                  processed | failed(+backoff) | dead
           → status='processing', RETURNING *)
```

**Componentes:**

| Componente | Tipo | Responsabilidad | Límite |
|---|---|---|---|
| `knowledge_claim_batch` | RPC SECDEF (0133) | Reclamar atómicamente un lote de eventos *due* (`status∈{pending,failed}` y `available_at<=now()`), pasarlos a `processing` con lock y devolverlos. | No ejecuta el efecto; no decide retry. |
| `knowledge_mark_processed` | RPC SECDEF (0133) | `processing → processed`, `processed_at=now()`. | — |
| `knowledge_mark_failed` | RPC SECDEF (0133) | Incrementar `retry_count`; si `<= MAX_RETRIES` → `failed` + `available_at` con backoff; si excede → `dead`. Devuelve el estado nuevo. | — |
| `knowledge_recover_stuck` | RPC SECDEF (0133) | Reponer a `failed` los eventos atascados en `processing` por más de un umbral (worker caído mid-process). | No borra; solo reprograma. |
| `drain.ts` | TS data layer | Bucle: claim → processEvent → mark; tope por `maxBatches`/`maxDuration`; devuelve resumen. | No expone datos sin auth; corre solo server-side (service_role). |
| `processEvent` | TS (interfaz + default no-op) | Punto de extensión del efecto de negocio. En E2.1: no-op `{ ok: true }`. | No implementa notificaciones (E3+). |
| `/api/knowledge/drain/route.ts` | Route handler | Auth fail-closed, `?dry`, invocar `drain.ts`, responder JSON. | No UI; no lógica de negocio. |
| `.github/workflows/knowledge-drain.yml` | Cron | Disparar el endpoint cada 5 min. | Solo dispara. |

## 3. Estados y ciclo de vida

**Valores del `CHECK` (ya existentes, 0126):** `pending`, `processing`, `processed`, `failed`, `dead`. (Tu "completed" = `processed`.)

```
            emit(p_status='pending')                 claim_batch (lock)
  (nace) ───────────────────────────► pending ──────────────────────────► processing
                                          ▲                                    │
                                          │                              processEvent()
                              mark_failed │ (retry<=MAX, backoff)              │
                                          │                          ┌─────────┴─────────┐
                                       failed ◄──────────────────────┤  error            │ ok
                                          │                          └─────────┬─────────┘
                       mark_failed (retry>MAX)                                 ▼
                                          ▼                                processed (terminal éxito)
                                        dead (terminal fallo)
   recover_stuck: processing (atascado > umbral) ──► failed (reprograma)
```

- **`pending`**: emitido para procesar async (no hay emisores `pending` en prod hoy; el worker se valida con fixtures controladas).
- **`processing`**: reclamado por un worker (lock); transitorio.
- **`processed`**: éxito terminal.
- **`failed`**: fallo transitorio reprogramado (`available_at` futuro por backoff).
- **`dead`**: fallo terminal tras `MAX_RETRIES`.

## 4. Contratos

### 4.1 Migración `0133_knowledge_dispatch` (RPCs SECDEF, hardened)

```sql
-- Reclamar lote atómico (concurrency-safe). batch default 50, lease default 5 min.
-- Al reclamar setea available_at = now()+lease: si el worker muere mid-process, el lease
-- vence y recover_stuck lo repone (sin columna nueva en knowledge_events => E1 intacto).
create or replace function public.knowledge_claim_batch(
  p_limit int default 50,
  p_lease interval default interval '5 minutes'
)
returns setof public.knowledge_events
language sql security definer set search_path = public, pg_temp as $$
  update public.knowledge_events ke
     set status = 'processing',
         available_at = now() + p_lease
   where ke.id in (
     select id from public.knowledge_events
      where status in ('pending','failed') and available_at <= now()
      order by available_at, seq
      limit greatest(p_limit, 0)
      for update skip locked
   )
   returning ke.*;
$$;

-- Marcar éxito.
create or replace function public.knowledge_mark_processed(p_id uuid)
returns void language sql security definer set search_path = public, pg_temp as $$
  update public.knowledge_events
     set status='processed', processed_at=now(), error=null
   where id = p_id and status='processing';
$$;

-- Marcar fallo con backoff exponencial (1→2→4 min) y dead tras MAX_RETRIES (default 3).
create or replace function public.knowledge_mark_failed(p_id uuid, p_error text, p_max_retries int default 3)
returns text language plpgsql security definer set search_path = public, pg_temp as $$
declare v_new int; v_status text;
begin
  update public.knowledge_events
     set retry_count = retry_count + 1
   where id = p_id and status='processing'
   returning retry_count into v_new;
  if v_new is null then return null; end if;   -- no estaba en processing (idempotente)
  if v_new <= p_max_retries then
    update public.knowledge_events
       set status='failed', error=left(p_error, 2000),
           available_at = now() + (power(2, v_new - 1) * interval '1 minute'),
           processed_at=null
     where id=p_id returning status into v_status;
  else
    update public.knowledge_events
       set status='dead', error=left(p_error,2000), processed_at=now()
     where id=p_id returning status into v_status;
  end if;
  return v_status;
end; $$;

-- Recuperar atascados: 'processing' cuyo lease venció (available_at < now()) => failed.
-- Usa available_at (seteado por claim) => NO requiere columna nueva en knowledge_events.
create or replace function public.knowledge_recover_stuck()
returns int language plpgsql security definer set search_path=public, pg_temp as $$
declare v_n int;
begin
  update public.knowledge_events
     set status='failed', available_at=now()
   where status='processing' and available_at < now();
  get diagnostics v_n = row_count; return v_n;
end; $$;

-- Hardening (las 4): revoke all from public; revoke execute from anon, authenticated; grant execute to service_role.
```

> **Backoff:** falla 1 → `retry_count=1` → +1 min · falla 2 → `2` → +2 min · falla 3 → `3` → +4 min · falla 4 → `>3` → `dead`. (Coincide con G7: MAX_RETRIES=3, 1→2→4.)

### 4.2 Endpoint `/api/knowledge/drain` (route handler)

## 5–9. Propiedades transversales

- **Reintentos/backoff (§5):** en `knowledge_mark_failed` (arriba). MAX_RETRIES=3; 1→2→4 min; luego `dead`.
- **Seguridad/RBAC (§6):** endpoint fail-closed por `CRON_SECRET`; RPCs SECDEF solo `service_role` (anon/authenticated revocados). El worker no expone datos; no requiere permiso de usuario (es máquina). No hay UI ⇒ sin `has_permission` de usuario en E2.1.
- **Idempotencia (§7):** `mark_*` actúan solo si `status='processing'` (re-llamadas no reprocesan); `claim_batch` con `skip locked` no entrega el mismo evento a dos workers; el emisor sigue con `on conflict do nothing`.
- **Locking (§7):** `FOR UPDATE SKIP LOCKED` en el subselect del claim (RE2-2: sin doble-procesamiento aun con crons solapados).
- **Recuperación ante fallos (§7):** `knowledge_recover_stuck` repone `processing` huérfano a `failed`; los `failed` se reintentan con backoff; los `dead` quedan para inspección manual.
- **Observabilidad (§8):** logging estructurado mínimo por lote (`component='knowledge_drain'`, `correlation_id` por corrida); el **catálogo EOL completo y la reconciliación TS↔SQL son E2.4** (no acá).
- **Performance (§9):** `claim_batch` usa `dispatch_idx (available_at, seq) WHERE status in ('pending','failed')` (O(log N)); lote acotado (`batchSize`, default 50); `maxDuration=60`, `maxBatches` para no exceder la ventana; sin escaneo completo.

## 11. Validaciones (read-only automáticas + smokes sin contaminar)

| Categoría | Verificación |
|---|---|
| **Estructura** | las 4 RPCs existen; ACL `{postgres, service_role}` (anon/auth sin execute); `0133` registrada. |
| **Claim/lifecycle (tx+rollback)** | emitir N `pending` (fixture) → `claim_batch` los pasa a `processing` y los devuelve → `mark_processed` → `processed`. |
| **Retry/backoff (tx+rollback)** | `mark_failed` 1ª vez → `failed`, `retry_count=1`, `available_at≈now+1min`; 2ª → +2min; 3ª → +4min; 4ª → `dead`. |
| **Locking** | dos `claim_batch` concurrentes no devuelven el mismo `id` (test con dos sesiones / `skip locked`). |
| **Idempotencia** | `mark_processed`/`mark_failed` sobre un evento que no está `processing` → no-op; re-claim no reprocesa. |
| **Recover stuck** | un `processing` "viejo" (fixture) → `recover_stuck` lo pasa a `failed`. |
| **Endpoint** | `curl` sin Bearer → 401; con Bearer → 200 + JSON; `?dry=1` no muta (conteo). |
| **Anti-leak** | tras los smokes: `knowledge_events`=152, 0 residuo, todo `processed`. |
| **Regresión** | backfill audit_log sigue idempotente (0); el emisor de `0132` intacto. |

## 12. Rollback

- **Rápido (sin DDL):** desactivar el cron (deshabilitar el workflow); el worker deja de correr; los eventos quedan en su estado. 
- **Completo:** `drop function` de las 4 RPCs de `0133`; revertir `route.ts`, `drain.ts` y el workflow (revert de código). `knowledge_events` y todo E1/E2.0 intactos. Eventos ya `processed` permanecen (append-only, correcto).
