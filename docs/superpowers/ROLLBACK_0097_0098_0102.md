## 1. Pre-condiciones para ejecutar el rollback

Antes de correr cualquier DROP, verificar:

1. **No hay datos en producción** que deban preservarse en las tablas de recon.
   - Si ya se ejecutaron conciliaciones reales, hacer backup selectivo:
     ```sql
     -- En SQL Editor de Supabase
     COPY (SELECT * FROM po_reconciliations) TO '/tmp/recon_backup.csv' CSV HEADER;
     COPY (SELECT * FROM po_reconciliation_diffs) TO '/tmp/recon_diffs_backup.csv' CSV HEADER;
     COPY (SELECT * FROM recon_events) TO '/tmp/recon_events_backup.csv' CSV HEADER;
     ```
     > O usar `supabase db dump` para snapshot completo antes de operar.

2. **Ninguna OC tiene status 'conciliada'** que dependa de las tablas de recon.
   La función `recon_approve` escribe `purchase_orders.status = 'conciliada'`.
   Si existen OCs en ese estado, el rollback no elimina ese dato (está en `purchase_orders`),
   pero el campo quedará en un estado inconsistente sin la tabla `po_reconciliations` que lo respalde.

   Verificar:
   ```sql
   SELECT COUNT(*) FROM purchase_orders WHERE status = 'conciliada';
   ```
   Si el resultado es > 0, decidir primero si revertir esas OCs a 'recibida' o 'aprobada'.

3. **El código de la app (branch feat/conciliacion-oc) NO está deployado** o se revertiría
   junto con este rollback. Las rutas `/compras/conciliacion/*` y el cliente
   `src/lib/recon/data.ts` usan `v_recon_status`, `recon_start`, `recon_approve`, etc.
   Si el código está en prod con las tablas eliminadas, la app arrojará errores 500.

4. **Las migraciones fiscales 0099/0100/0101 se aplican/no-aplican independientemente.**
   Este rollback no afecta ni requiere rollback de las migraciones fiscales.

## 2. SQL de rollback — en orden de dependencia inverso

Ejecutar en el SQL Editor de Supabase prod (`arsksytgdnzukbmfgkju`), en este orden exacto:

### Paso 1 — Eliminar vista (0102)

```sql
-- v_recon_status depende de po_reconciliations, purchase_orders, supplier_invoices,
-- po_reconciliation_diffs. Debe eliminarse antes que las tablas.
DROP VIEW IF EXISTS v_recon_status;
```

### Paso 2 — Eliminar RPCs / funciones (0098)

```sql
-- RPCs de escritura (SECURITY DEFINER)
DROP FUNCTION IF EXISTS recon_start(uuid, uuid, smallint, jsonb) CASCADE;
DROP FUNCTION IF EXISTS recon_approve(uuid, text) CASCADE;
DROP FUNCTION IF EXISTS recon_reject(uuid, text) CASCADE;
DROP FUNCTION IF EXISTS recon_accept_diff(uuid, text) CASCADE;
DROP FUNCTION IF EXISTS recon_add_note(uuid, text) CASCADE;
DROP FUNCTION IF EXISTS recon_send_to_review(uuid, text) CASCADE;

-- Función auxiliar de rol (usada internamente por las RPCs anteriores)
-- Eliminar DESPUÉS de las que la invocan para que CASCADE no sea necesario
DROP FUNCTION IF EXISTS _recon_assert_role() CASCADE;
```

### Paso 3 — Eliminar trigger y su función (0097)

```sql
-- Primero el trigger (queda huérfano si se elimina la función antes)
DROP TRIGGER IF EXISTS tg_recon_events_no_delete ON recon_events;

-- Luego la función de trigger
DROP FUNCTION IF EXISTS tg_forbid_delete_recon_events() CASCADE;
```

### Paso 4 — Eliminar políticas RLS (0097)

Las políticas se eliminan automáticamente con DROP TABLE, pero si se quiere
limpiar de forma explícita antes:

```sql
DROP POLICY IF EXISTS "recon_read"       ON po_reconciliations;
DROP POLICY IF EXISTS "recon_diffs_read" ON po_reconciliation_diffs;
DROP POLICY IF EXISTS "recon_events_read" ON recon_events;
```

### Paso 5 — Eliminar tablas en orden FK inverso (0097)

```sql
-- recon_events y po_reconciliation_diffs referencian po_reconciliations
-- Eliminar hijas primero
DROP TABLE IF EXISTS recon_events              CASCADE;
DROP TABLE IF EXISTS po_reconciliation_diffs   CASCADE;

-- Tabla header (referenciada por las dos anteriores)
DROP TABLE IF EXISTS po_reconciliations        CASCADE;
```

> **Nota sobre CASCADE:** los índices `idx_recon_po`, `idx_recon_inv`, `idx_recon_status`,
> `idx_recon_diffs`, `idx_recon_events` se eliminan automáticamente al hacer DROP TABLE.

### Paso 6 — Eliminar tipos (0097)

```sql
-- Los tipos deben eliminarse después de las tablas que los usan
DROP TYPE IF EXISTS recon_diff_field_t CASCADE;
DROP TYPE IF EXISTS recon_status_t     CASCADE;
```

### Paso 7 — Revertir columna supplier_invoice_id en purchase_orders (0097)

```sql
-- La migración 0097 agregó esta columna a purchase_orders.
-- El rollback debe eliminarla para dejar la tabla en su estado previo.
-- PRECAUCIÓN: si ya hay OCs con supplier_invoice_id != NULL,
-- esos UUIDs se perderán. Verificar primero:
--   SELECT COUNT(*) FROM purchase_orders WHERE supplier_invoice_id IS NOT NULL;
ALTER TABLE purchase_orders DROP COLUMN IF EXISTS supplier_invoice_id;
```

> **Alerta:** `recon_status_t` es usado también en la función `tg_forbid_delete_recon_events`
> y en los argumentos de `recon_events`. Al haber eliminado todo lo anterior, el CASCADE
> es seguro pero innecesario. Si aparece un error "type in use", verificar que los pasos
> anteriores se completaron.

### Script completo en bloque único (para ejecutar de una vez)

```sql
-- ============================================================
-- ROLLBACK 0097 + 0098 + 0102 — Conciliación de OC
-- Ejecutar en SQL Editor Supabase prod (arsksytgdnzukbmfgkju)
-- PRECONDICIÓN: verificar que no hay OCs con status='conciliada'
-- ============================================================

-- Paso 1: Vista
DROP VIEW IF EXISTS v_recon_status;

-- Paso 2: RPCs
DROP FUNCTION IF EXISTS recon_start(uuid, uuid, smallint, jsonb) CASCADE;
DROP FUNCTION IF EXISTS recon_approve(uuid, text) CASCADE;
DROP FUNCTION IF EXISTS recon_reject(uuid, text) CASCADE;
DROP FUNCTION IF EXISTS recon_accept_diff(uuid, text) CASCADE;
DROP FUNCTION IF EXISTS recon_add_note(uuid, text) CASCADE;
DROP FUNCTION IF EXISTS recon_send_to_review(uuid, text) CASCADE;
DROP FUNCTION IF EXISTS _recon_assert_role() CASCADE;

-- Paso 3: Trigger
DROP TRIGGER IF EXISTS tg_recon_events_no_delete ON recon_events;
DROP FUNCTION IF EXISTS tg_forbid_delete_recon_events() CASCADE;

-- Paso 4: Políticas (limpieza explícita)
DROP POLICY IF EXISTS "recon_read"        ON po_reconciliations;
DROP POLICY IF EXISTS "recon_diffs_read"  ON po_reconciliation_diffs;
DROP POLICY IF EXISTS "recon_events_read" ON recon_events;

-- Paso 5: Tablas (orden FK inverso)
DROP TABLE IF EXISTS recon_events             CASCADE;
DROP TABLE IF EXISTS po_reconciliation_diffs  CASCADE;
DROP TABLE IF EXISTS po_reconciliations       CASCADE;

-- Paso 6: Tipos
DROP TYPE IF EXISTS recon_diff_field_t CASCADE;
DROP TYPE IF EXISTS recon_status_t     CASCADE;

-- Paso 7: Columna supplier_invoice_id en purchase_orders
-- VERIFICAR ANTES: SELECT COUNT(*) FROM purchase_orders WHERE supplier_invoice_id IS NOT NULL;
ALTER TABLE purchase_orders DROP COLUMN IF EXISTS supplier_invoice_id;

-- ============================================================
-- FIN DEL ROLLBACK
-- ============================================================
```

## 4. Dependencias analizadas — por qué el rollback es seguro

### Entre recon y fiscal (0099/0100/0101)

| Tipo de dependencia | ¿Existe entre recon y fiscal? | Detalle |
|---|---|---|
| Tablas / JOINs | NO | Las tablas recon referencian `purchase_orders` y `supplier_invoices` (tablas base pre-existentes), no las tablas fiscales (`ganancias_retenciones`, `ganancias_retention_params`, etc.) |
| Vistas | NO | `v_recon_status` solo junta `po_reconciliations`, `purchase_orders`, `supplier_invoices` y `po_reconciliation_diffs` |
| Funciones / RPCs | NO | Las RPCs de recon (`recon_start`, etc.) no invocan ninguna función fiscal. Las RPCs fiscales (`ap_upsert_retencion_ganancias`, `ap_get_retencion_context`, etc.) no invocan ninguna función recon |
| Triggers | NO | El único trigger de recon (`tg_recon_events_no_delete`) opera solo sobre `recon_events` |
| Enums / tipos | NO | `recon_status_t` y `recon_diff_field_t` son tipos propios del módulo recon. Los fiscales no tienen tipos enum propios |
| RLS policies | NO | Las policies de recon no referencian tablas fiscales y viceversa |
| Índices | NO | Los índices son internos a cada módulo |
| FK cruzadas | NO | `ganancias_retenciones` referencia `supplier_invoices` (tabla base), no tablas recon. `po_reconciliations` referencia `purchase_orders` y `supplier_invoices` (tablas base), no tablas fiscales |

**Conclusión:** el único elemento compartido entre ambos módulos es la tabla base
`supplier_invoices` (pre-existente, no creada por ninguno de los dos módulos).
Eliminar recon no afecta fiscal, y viceversa.

### Dependencias de recon sobre tablas base (pre-existentes en prod)

Estas dependencias NO se rompen con el rollback (las tablas base permanecen):

- `po_reconciliations.purchase_order_id → purchase_orders(id)` — tabla de OC existe desde migración 0008
- `po_reconciliations.supplier_invoice_id → supplier_invoices(id)` — tabla existe desde migración 0014
- `po_reconciliations.initiated_by → auth.users(id)` — tabla de auth de Supabase, siempre presente
- `recon_approve()` escribe en `purchase_orders.status` y `purchase_orders.factura_id` — estos campos preexisten en la tabla base

## 5. Advertencia: módulos dependientes del recon

### 5.1 Tesorería — campo `listo_para_pago`

La vista `v_recon_status` expone el campo `listo_para_pago` (booleano: `status IN ('conciliada','con_diferencias')`).

El código de la app en `src/lib/recon/data.ts` (branch `feat/conciliacion-oc`) usa este campo
para decidir si una OC puede habilitarse para pago.

Si se hace rollback de la BD pero el código del módulo recon sigue deployado, **las rutas
de Conciliación fallarán** con error de tabla/vista no encontrada.

**Verificación post-rollback:**
```bash
# Verificar que las rutas de conciliación no están activas en prod
curl -s https://nexus.logisticatops.com/compras/conciliacion \
  -H "Cookie: ..." | grep -c "500\|Error"
```

Si el rollback de código no se hace en simultáneo con el de BD, agregar una guardia en
el código que devuelva lista vacía ante error de "relation does not exist".

### 5.2 `purchase_orders.status = 'conciliada'`

El valor `'conciliada'` en el enum `po_status_t` pre-existe desde migración 0008.
El rollback de recon NO elimina ese valor del enum (sería un DDL separado y breaking).
Las OCs que llegaron a ese estado quedan en `'conciliada'` pero sin `po_reconciliations`
que las respalde. Es un estado inconsistente que requiere corrección manual si aplica.

### 5.3 Migraciones fiscales (0099/0100/0101) — NO afectadas

Las migraciones fiscales pueden aplicarse o revertirse con total independencia
del rollback de recon. No hay riesgo cruzado.
