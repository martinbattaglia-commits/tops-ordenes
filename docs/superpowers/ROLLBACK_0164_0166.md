# ROLLBACK — Migraciones F4.2 `0164`–`0166` (Centro de Incidentes)

> Procedimiento de reversa COMPLETA de la capa DB de F4.2. Ejecutar SOLO con
> autorización expresa de Dirección, a mano en el SQL Editor de prod
> (`arsksytgdnzukbmfgkju`), en el ORDEN indicado (inverso al apply: 0166 → 0165 → 0164).
> Ninguna migración de F4.2 modifica objetos previos de forma destructiva: el
> rollback es un DROP limpio de objetos nuevos + limpieza de filas de catálogo.

## Consideraciones previas

- **Datos**: si ya existen incidentes reales, `drop table connect_incidents` los
  destruye. Evaluar `create table connect_incidents_bak as select * from connect_incidents;`
  antes (o export CSV). Las conversaciones `kind='incident'` NO se borran (son
  conversaciones normales del motor de chat; el valor de enum `incident` es de 0143
  y NO se toca).
- **Apéndices que permanecen a propósito** (append-only / inofensivos):
  filas de `audit_log` (`entity='connect_incident'`), notificaciones ya emitidas
  (`kind='connect_incident'` — el front las rutea a una página que dejaría de
  existir tras revertir la UI; siguen abriendo `/connect/notificaciones` vía fallback),
  eventos `knowledge_events` si la fuente se hubiera habilitado (no es el caso: D5 la
  deja `enabled=false`).
- El frontend correspondiente se revierte por deploy del commit anterior
  (`bef2f78`); la DB y el front son independientes (el front sin migraciones
  degrada con gracia: lista vacía + acciones fallan con mensaje).

## 3. Revertir 0164 (schema + valor de enum)

```sql
-- ⚠️ El valor 'incident_admin' agregado a permission_action_t NO se puede
-- dropear (Postgres no soporta DROP VALUE). Queda como RESIDUO ACEPTADO:
-- inofensivo (ninguna fila lo usa tras el delete de arriba).

-- Realtime (idempotente; ignora si no estaba publicada).
do $$ begin
  execute 'alter publication supabase_realtime drop table public.connect_incidents';
exception when undefined_object or undefined_table then null; end $$;

-- Tabla (⚠️ destruye datos de incidentes — ver Consideraciones) + triggers/índices (caen con la tabla).
drop table if exists public.connect_incidents;

-- Funciones y secuencia propias.
drop function if exists public._connect_set_incident_public_id();
drop sequence if exists public.connect_incident_seq;

-- Enums (solo usados por connect_incidents; sin dependientes tras el drop).
drop type if exists public.connect_incident_status_t;
drop type if exists public.connect_incident_severity_t;

notify pgrst, 'reload schema';
```
